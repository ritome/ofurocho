## 1. アプリの正体

- **正式名称**：お風呂帖（おふろちょう）
- **ホーム画面表示**：お風呂帖
- **作者**：ヒトノワスタジオ（HITONOWA STUDIO）
- **位置づけ**：第2作（第1作は「ワタシノヒトリゴト」）
- **用途**：居住型ケア施設の入浴記録を業務用に管理する単一HTMLアプリ
- **設計の北極星**：**スタッフが画面を見て1秒で「これだけタップすればいい」と分かる**こと

スタッフ専用、施設のiPad 1台運用が第1段階。Safari起動 file:// 経由でも動くこと。

詳細仕様はすべて `docs/ofurocho-handoff.md` に集約。読み合わせる順序: ハンドオフ書 → このCLAUDE.md。

## 2. 設計思想（やる／やらない）

### やる
- 業務用だがプレッシャーゼロ（書き忘れに優しい）
- 山吹（D3）系の北欧テイストで上品に
- データの主権はユーザーに（バックアップは手動JSON、Excel取り込みは月初手動）
- 単一HTML完結、ビルド不要、CDN経由のReact 18 + Babel + SheetJS

### やらない（重要）
- **外部にデータを送信しない**。サーバー、クラウド、analytics、トラッキングすべて禁止
- **CDN以外の外部依存を入れない**。React・SheetJSのCDNは許容、それ以外は要相談
- **山吹系の配色を崩さない**。新色追加・既存色変更は要相談
- **スタッフ画面に新しい操作を増やさない**。複雑さは予定編集モードと設定タブへ
- **記録テーブルを別に持たない**。bathPlans エントリの「状態」が「予定」⇄「済」に遷移するモデル
- **TypeScriptやビルドツールを入れない**。Vanilla + Babel スタンドアロン縛り
- **600以上のフォントウェイトを使わない**。400/500の2段階のみ
- **600 / 700 のような太字を使わない**

## 3. 技術スタック

- **形式**：単一 `index.html` ファイル（JS/CSSは内部に直書きでOK、CDN以外の外部参照禁止）
- **フレームワーク**：React 18 + Babel スタンドアロン（CDN経由）
- **データ保存**：localStorage（キー名は `ofurocho.{エンティティ}`）
- **Excel I/O**：SheetJS（CDN）
- **対象ブラウザ**：iPad Safari（主）、Mac/Windowsモダンブラウザ（副）

CDN（`docs/ofurocho-handoff.md` §1 参照）:
```html
<script crossorigin src="https://unpkg.com/react@18/umd/react.production.min.js"></script>
<script crossorigin src="https://unpkg.com/react-dom@18/umd/react-dom.production.min.js"></script>
<script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
<script src="https://cdn.sheetjs.com/xlsx-latest/package/dist/xlsx.full.min.js"></script>
```

## 5. データ構造（要約）

すべて `localStorage`、キーは `ofurocho.{エンティティ}`。詳細スキーマは `docs/ofurocho-handoff.md` §3。

| キー | 型 | 内容 |
|---|---|---|
| `ofurocho.residents` | array | 利用者（id, 名前, フリガナ, 部屋番号, 区分, 入所日, 在籍中, 備考） |
| `ofurocho.staff` | array | スタッフ（id, 名前, 職種, サブ役割[], 在籍中） |
| `ofurocho.bathPlans` | array | 入浴予定/記録（id, 日付, セッション AM/PM, 利用者ID, 状態 予定/済, 記録時刻, 体制ID） |
| `ofurocho.sessionTeams` | array | 体制（id, 日付, セッション, 4ポジション割当, 作成時刻） |
| `ofurocho.settings` | object | 施設名, 入浴目標, 警告レベル, バージョン, 最終バックアップ |

### 互換性ルール（重要）

- 新フィールドを追加するときは初期値（デフォルト）を必ず用意する。古いバックアップを読んで欠損していてもクラッシュしないこと
- 削除はやめる。しばらく参照しないだけにして、データは残す
- スキーマの破壊変更はキー名を `ofurocho.v2.*` のようにバージョン分岐させる

## 9. 既知の判断・運用メモ

### React + Babel スタンドアロン採用の理由

ヒトリゴトはVanilla JSで通したが、お風呂帖はタブ・モーダル・ドロップダウン多数で状態管理が複雑になるためReactを導入。ただしビルドは入れない（Babel スタンドアロン）。コードは `<script type="text/babel">` 内に書く。

### 「予定」と「記録」を分けない理由

`bathPlans` の単一エンティティで、「状態」フィールドが `予定` ⇄ `済` に遷移するだけ。記録テーブルを別に持つと整合性管理が増える。スタッフ画面は「今日の予定をタップして済にする」だけなのでこのモデルで十分。

### 体制データの位置づけ

`sessionTeams` は「誰がそのセッションを担当したか」のメタ情報。bathPlans の `体制ID` から参照される。集計時には参照されるが、入浴記録の本質ではない。

### 体制と記録の時間独立性（重要）

**実運用では、体制が決まる前に入浴を始めることもある**（看護師の出勤確定が遅れる等）。よって設計：

1. **カードタップは体制の有無を問わず常に通る**。プロンプトを出さない
2. **体制保存時、同日同セッションの既存記録に遡って `体制ID` を紐付ける**
   - 対象：その日のそのセッションで `体制ID` が `null` または保存される体制と同じID の記録
   - 別の体制IDが既に振られている記録（過去のセッションで確定したもの）は触らない
3. **フッターの未設定時メッセージは催促しない**。「あとで設定できます（記録には影響しません）」

**なぜ重要**：先に記録だけしておいて後で体制を入力する流れでも、ちゃんと「誰が担当した記録か」が残る。これが無いと、体制を後から入れた時に過去の記録だけが「担当不明」のまま残ってしまう。

実装ポイント：`handleSaveTeam` で保存時に `bathPlans` をマップして紐付け。引き継ぎ書 §4.1 の「ソフトプロンプト→体制設定画面へ」は本実装では採用しない（仕様変更）。

### 警告色の使い分け

- `--warn-red` (#b8503a)：5日以上未浴／達成率33%未満／削除ボタンなど
- `--accent-deep` (#9a7232)：3〜4日未浴／午後セッション／達成率注意域
- 派手な赤や黄色のアラートは使わない

### XSS 対策

利用者名・スタッフ名・備考のテキスト挿入は React の JSX が自動的にエスケープするので安全。`dangerouslySetInnerHTML` は使わない。Excel取り込み時もパースしてオブジェクト化するのみ。

### iPad Safari file:// の制約

- `URL.createObjectURL` は使えるが `Blob` のダウンロードは挙動が不安定 → `download` 属性を持つ `<a>` クリック方式で実装、必要ならテキスト表示フォールバック
- localStorage は file:// でも各オリジン（パス）ごとに分かれて保存される
- Service Worker は file:// では動かない → PWA化はhttps配信が前提

### 開発モードの切替

URLに `?dev=1` を付けて起動すると、開発ツールバー（サンプル投入／本日サンプル予定／過去60日履歴／全データ削除）が表示される。**永続化はしない**：パラメータなしのURLでアクセスすれば常に本番モード（バー非表示）。これは「本番URLを開いた時に必ず本番として動く」ことを保証する判断。過去バージョンに残った `localStorage["ofurocho.devMode"]` は起動時に自動でクリーンアップする。

### 初回起動フロー

利用者・スタッフが両方0件のときは、入浴記録／記録一覧タブで Welcome 画面を表示し「設定タブで Excel 取り込み」を促す。設定タブには切り替え可能（取り込み作業中はWelcomeが邪魔にならない）。開発モード時のみ「サンプルデータで試す」ボタンを併記。

### 印刷の方針

- 入浴予定表（編集モード→「予定を印刷」）：A4縦、午前/午後/両方を選べる。両方は2ページに分割
- 達成率レポート（記録一覧→「印刷」）：週・月はA4縦、半年・年はA4横（自動）
- すべて画面UI（タブ・フィルタ・ボタン・モーダル）を `.no-print` で隠し、印刷専用テンプレートを `.print-only` で表示
- 白黒プリンタを想定し、達成率バーや朝/午バッジはグレー濃淡で識別可能に

## 10. 今後の判断指針

仕様追加が来たとき、以下に照らして判断：

1. **スタッフのタップが増えないか**（増えるなら却下寄り、編集モードか設定へ）
2. **データを端末外に出さないで実現できるか**（できなければ却下）
3. **山吹系の北欧テイストを保てるか**（新色・派手な要素は要相談）
4. **CDN以外の依存追加なしで実装できるか**（できなければ要相談）
5. **既存データとの互換性を保てるか**（デフォルト値の追加で済むか）
